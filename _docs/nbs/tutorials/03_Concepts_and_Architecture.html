Core Concepts and Architecture

Core Concepts and Architecture

This tutorial introduces the fundamental concepts and architecture of the ML Deploy platform. Understanding these concepts is essential for working effectively with the platform and implementing ML deployment workflows.

Platform Architecture Overview

The ML Deploy platform is built around a contract-based architecture that ensures consistency and reliability across different deployment scenarios. The architecture is designed to be specification-first, meaning that specifications are defined and ratified before implementation begins.

Key Architectural Principles

  1. Separation of Concerns: Clear boundaries between data management, model training, deployment, and infrastructure
  2. Contract-Based Design: Well-defined interfaces between components
  3. Immutability: Notebook source code remains immutable; parameters are injected externally
  4. Traceability: Complete tracking of data, models, and deployments
  5. Reproducibility: Deterministic builds and execution environments

Core Concepts

1. Execution Patterns (EX-01 → EX-03)

The foundation of ML Deploy is built on three core execution patterns that form the basis for all ML workflows:

EX-01: Training with Traceability

What it is: A complete training workflow that captures comprehensive metadata about the training process.

Why it matters: Ensures every training run is fully traceable and reproducible, which is essential for debugging, auditing, and compliance.

Key components: - Model training execution - MLflow experiment tracking - Comprehensive metadata capture - Training artifact storage - Complete audit trail

Example:

# Training with complete traceability
with mlflow.start_run(run_name="experiment_1") as run:
    # Set traceability tags
    mlflow.set_tag("dataset_version", "v1.0")
    mlflow.set_tag("feature_revision", "engineered_v2")
    mlflow.set_tag("model_type", "xgboost")
    
    # Log hyperparameters
    mlflow.log_params({"max_depth": 6, "learning_rate": 0.1})
    
    # Execute training
    model = train_model(X_train, y_train)
    
    # Log model and metrics
    mlflow.xgboost.log_model(model, "model")
    mlflow.log_metrics({"accuracy": 0.95})

EX-02: Artifact Bundling and Versioning

What it is: The process of packaging trained models and associated artifacts into versioned, deployment-ready bundles.

Why it matters: Creates portable, versioned model artifacts that can be reliably deployed across different environments.

Key components: - Model serialization - Version registration in MLflow - Artifact bundling - Semantic versioning - Immutable references

Example:

# Create model artifact bundle
model_artifact = ModelArtifact(
    model_path=trained_model_path,
    scaler_path=trained_scaler_path,
    metadata={
        "version": "v1.0",
        "dataset_version": "v1.0",
        "feature_revision": "engineered_v2",
        "hyperparameters": {"max_depth": 6}
    }
)

# Register in MLflow registry
registration_result = register_model_artifact(
    artifact_bundle_path=model_artifact.to_bundle(),
    model_name="diabetes_predictor",
    model_version="v1.0"
)

EX-03: Local Deployment and Serving

What it is: Local model deployment using nbdev/Python for inference with comprehensive prediction logging.

Why it matters: Enables local testing and validation of model artifacts in a controlled environment before production deployment.

Key components: - Local model loading and inference - Prediction logging in JSONL format - Deployment metadata tracking - Local serving endpoints - Performance monitoring

Example:

# Local model serving
local_serving = LocalModelServing(model_bundle_path)
prediction_logger = PredictionLogger(log_dir)

# Make predictions with logging
result = local_serving.predict(input_data)

# Log prediction
prediction_logger.log_prediction(
    result,
    {"model_version": "v1.0", "timestamp": now()}
)

2. Contract-Based Architecture

What it is: A design approach where components interact through well-defined interfaces rather than direct implementation dependencies.

Why it matters: Ensures consistency, enables testing, and allows components to be swapped or upgraded independently.

Key Contracts

Web UI Contracts
  • NotebookExecutionRequest: Standard format for notebook execution requests
  • RunVisibility: Controls output sharing and access control
  • ExecutionOrchestrator: Manages execution across different backends
# Example contract usage
request = NotebookExecutionRequest(
    notebook_ref="abc123",
    notebook_path=Path("/path/to/notebook.ipynb"),
    parameters={"learning_rate": 0.1},
    environment="dev",
    resource_requirements={"cpu": 2, "memory": "4GB"}
)

orchestrator = ExecutionOrchestrator()
result = orchestrator.submit_execution(request)
Execution Backend Contracts
  • LocalExecutionAdapter: Maps requests to local execution
  • SlurmExecutionAdapter: Maps requests to Slurm execution
  • KubernetesExecutionAdapter: Maps requests to Kubernetes execution
# Backend abstraction
backend = get_execution_backend(request.environment)
result = backend.submit(request)

3. Governance Framework

What it is: A system of controls and processes that ensures ML operations meet quality and compliance standards.

Why it matters: Provides structure for development, ensures quality, and enables auditability.

Key Governance Components

Repository Governance State
@dataclass(frozen=True)
class RepositoryGovernanceState:
    phase: RepositoryPhase = "specification_first"
    spec_quality_gate_passed: bool = False
Phase Transitions
  • specification_first: Focus on defining specifications and architecture
  • implementation_enabled: Allow production-ready implementation work
# Phase transition example
state = RepositoryGovernanceState(
    phase="specification_first",
    spec_quality_gate_passed=True
)

decision = request_implementation_transition(
    state=state,
    explicit_user_confirmation=True
)

if decision.allowed:
    print("Implementation can proceed!")

4. Infrastructure as Code

What it is: Managing infrastructure through code rather than manual configuration.

Why it matters: Enables version control, reproducibility, and automation of infrastructure management.

Key Infrastructure Components

Nix Containerization
  • No Docker files: All containers defined via Nix dockerTools
  • Docker-first reproducibility: Containers built via Nix but run as standard Docker images
  • Immutable references: Pinned package versions for reproducibility
# Example Nix container definition
{ pkgs, ... }:

pkgs.dockerTools.buildImage {
  name = "mlflow-training";
  tag = "latest";
  
  buildInputs = with pkgs; [
    python311
    mlflow
    xgboost
  ];
  
  config = {
    Cmd = ["/bin/bash"];
    WorkingDir = "/app";
    ExposedPorts = {"5000/tcp" = {}};
  };
}
Terranix/OpenTofu
  • Infrastructure-as-code: Declarative infrastructure provisioning
  • Multi-profile support: Both local_emulation and cloud deployment
  • Version control: Infrastructure definitions tracked in git
# Example Terranix configuration
{ config, lib, ... }:

{
  imports = [ ./shared.nix ];
  
  config.mlDeploy = {
    profile = "local_emulation";
    mlflowTrackingUri = "http://localhost:5001";
    awsEndpointUrl = "http://localhost:4566";
  };
}

5. Data Lineage

What it is: Complete tracking of data flow through the ML pipeline, from source to output.

Why it matters: Ensures traceability, enables debugging, and supports compliance requirements.

Lineage Components

Dataset Tracking
  • Source identification: Where data comes from
  • Version control: How data changes over time
  • Provenance: Complete history of data transformations
Model Training Lineage
  • Input data sources: What data was used for training
  • Feature engineering: How features were created
  • Hyperparameters: What parameters were used
  • Performance metrics: How well the model performed
Deployment Lineage
  • Model version: Which version of the model was deployed
  • Deployment environment: Where and when it was deployed
  • Performance monitoring: How it’s performing in production

6. Multi-Topology Support

What it is: The ability to deploy and run ML workflows in different environments and scales.

Why it matters: Supports different use cases from local development to production deployment.

Supported Topologies

Local Development Topology
  • Use case: Development and testing
  • Components: Local execution, MLflow tracking, containerized dependencies
  • Benefits: Fast iteration, local reproducibility
Distributed Training Topology
  • Use case: Large-scale model training
  • Components: Lambda.ai, Slurm coordination, distributed data loading
  • Benefits: Scalability, fault tolerance, job management
Batch Inference Topology
  • Use case: Large-scale batch predictions
  • Components: Kubernetes batch jobs, model serving, output lineage
  • Benefits: Efficiency, cost optimization, traceability
Online Inference Topology
  • Use case: Real-time predictions
  • Components: Online serving, deployment rollbacks, monitoring
  • Benefits: Low latency, reliability, rollback capability

Architecture Decision Records

Why Architecture Decisions Matter

Architecture decisions ensure consistency, guide implementation, and document rationale for technical choices. They help:

  1. Maintain consistency across the platform
  2. Guide implementation work
  3. Document rationale for technical choices
  4. Onboard new team members quickly
  5. Support decision-making for future changes

Key Architecture Decisions

1. Nix-Only Containerization

Decision: No Docker files; all containers defined via Nix dockerTools Rationale: Reproducibility, version control, and Docker-first compatibility Implications: All container definitions must be Nix-based

2. Contract-Based Design

Decision: Well-defined interfaces between components Rationale: Consistency, testability, and component independence Implications: All components must implement defined contracts

3. Specification-First Development

Decision: Specifications before implementation Rationale: Quality assurance, architectural consistency Implications: Implementation work requires specification ratification

4. Multi-Profile Infrastructure

Decision: Support for local_emulation and cloud deployment Rationale: Development needs, production parity Implications: Infrastructure code must support multiple profiles

5. Immutable Notebooks

Decision: Notebook source code remains immutable Rationale: Reproducibility, version control Implications: Execution parameters must be injected externally

# Complete workflow example
def complete_ml_workflow(work_dir: Path):

Common Questions

Q: Why use contracts instead of direct implementation?

A: Contracts ensure consistency, enable testing, and allow components to be swapped independently. They provide clear interfaces that guide implementation while allowing flexibility in how those interfaces are implemented.

Q: What’s the difference between EX-01, EX-02, and EX-03?

A: EX-01 focuses on training with traceability, EX-02 on model packaging and versioning, and EX-03 on local deployment and serving. Together they form a complete workflow from training to deployment.

Q: Why Nix instead of Docker?

A: Nix provides better reproducibility, version control, and layer optimization while maintaining Docker-first compatibility. It eliminates Dockerfiles while still producing standard Docker images.

Q: How does governance affect development?

A: Governance ensures that implementation work meets quality standards and follows proper processes. It prevents premature implementation and ensures architectural consistency.
